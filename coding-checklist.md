# 🧩 신입 개발자가 반드시 습관화해야 할 부분들
## ✅ 1. 요구사항 & 커뮤니케이션(일의 시작)
### 핵심 키워드 : 이해 없이 코딩 금지
    - 요구사항을 문서로 정리하고 모호한 부분은 반드시 질문
    - “왜 이 기능이 필요한가?”를 이해
    - 입력/출력 데이터 정의를 명확히 확인(Format 포함)
    - 회의/대화 후에는 반드시 메모 기록(나중에 기억에 의존하지 않도록)
    - API/서비스 간 "계약(Interface)" 정확히 파악
    - "내가 이해한 내용이 맞는지" 재확인
    - "당연하다"라고 느껴지는 부분일수록 질문
## ✅ 2. 사고 습관(개발자의 기본 태도)
### 핵심 키워드 : 의심 + 검증
    - 데이터는 항상 의심(값/형식/흐름)
    - "왜 이 값이 나오지?" 생각
    - "에러 없는데 안 되는 상황"을 특히 경계
    - 흐름을 머릿속이 아니라 구조로 이해
    👉🏻 반드시 가져야 할 마인드
        - "안 되는 이유는 반드시 존재한다"
        - "못 찾은 것뿐이다"
## ✅ 3. 코드 작성 습관
### 핵심 키워드 : 읽기 쉬움 + 유지보수성
    - 팀 코드 컨벤션 준수
    - 변수/메서드 이름을 의미 있게 작성
    - 주석은 “왜”를 설명하고, “무엇”은 코드로 작성
    - 작은 단위로 커밋(한 번에 많은 변경사항 커밋 금지)
    - 재사용 가능한 코드 작성
    - 코드 복사+붙여넣기 전에 반드시 이해
    - 변환 로직(날짜, 문자열 등)은 따로 분리
    - 중복 코드 제거(한 번만 작성)
## ✅ 4. 데이터 처리 & 서비스 호출
### 핵심 키워드 : 입출력 명세/검증/추적
    - Input/Output 타입과 포맷 항상 확인
    - 날짜/시간 포맷(yyyyMMdd vs yyyyMMddHHmmss) 확인
    - null/빈 값 처리 필수(validation 습관화)
    - 필수 파라미터 누락 방지
    - 단위/타임존 차이 확인
    - 값이 넘어가는 전체 흐름 추적
    - "자동 변환(프레임워크 간 변환 등)" 절대 신뢰 금지
    - 변환 결과를 눈으로 확인(로그 등)
## ✅ 5. 테스트 & 검증 습관
### 핵심 키워드 : 깨지는 걸 먼저 확인
    - 단위 테스트 작성
    - 정상 케이스 + 경계값 케이스 + 에러 케이스 모두 테스트
    - 코드 변경 시 회귀(리그레이션) 테스트 실행
    - 로컬에서 충분히 검증 후 배포
    - null 넣어보기
    - 빈 값("") 넣어보기
    - 이상한 값 일부러 넣어보기
    👉🏻 핵심 문장
        "잘 되는 테스트가 아니라 실패하는 테스트를 만든다"
## ✅ 6. 디버깅 & 로그 습관
### 핵심 키워드 : 눈으로 보는 개발
    - 로그를 꼼꼼히 남기기
    - 로그 읽는 습관
    - 입력/변환/출력 지점마다 로그 확인
    👉🏻 반드시 로그를 남겨야 하는 위치
        - 서비스 호출 전
        - 변환 직후
        - 외부 API 호출 전/후
        👉🏻 핵심
            "확인하지 않은 값을 절대 신뢰하지 말라"
## ✅ 7. 운영 & 장애 대응
### 핵심 키워드 : 실무 생존 능력
    - 배포 환경 설정 차이 항상 의식
    - 장애 발생 시 원인 분석 과정 기록
    - 로그 기반으로 문제 추적
    - 성능 문제를 사전에 고려
    - 문제 발생 시 흐름을 구간별로 나눠서 순차적으로 확인
        - 흐름 구간 구분 : 요청 진입 → 파싱 → 비즈니스 로직 → 외부 호출 → DB → 응답
    - 최근 변경 사항부터 의심
## ✅ 8. 보안 & 안정성
### 핵심 키워드 : 기본적인 안전 장치
    - 입력값 검증(SQL Injection, XSS)
    - 인증/권한 체크 포함
    - 개인정보 마스킹/암호화
    - 외부 API 호출 시 보안 규칙 준수
    👉🏻 실무 핵심
        "사용자가 이상한 값을 넣는다는 전제"
## ✅ 9. 협업 & 성장
### 핵심 키워드 : 개발자는 혼자 일하는 직업이 아님
    - 코드 리뷰 적극 참여
    - 팀 룰/컨벤션 존중
    - 문서화 습관(README, API 명세 등)
    - 학습한 내용 공유

    - 기능은 작게 나누어 개발
    - 각 단계에서 바로 검증
    - 문제를 작게 쪼개서 해결

## 우선 적용 권장 순서 (우선순위 1→4)
1. 요구사항 확인 습관 (요청 즉시, 코딩 시작 전 바로 수행)
    - 행동 : 요구사항 문서화, 입력/출력 포맷 표준화, 이해한 내용 이메일로 재확인.
	- 목표 : 첫 주 내 모든 티켓에 요구사항 체크리스트 첨부. 
2. 테스트 우선 습관 (매 단계 병행)
	- 행동 : 기능 개발 전 실패하는 테스트(🔴) 작성 → 코드 구현 → 통과(🟢).
	- 목표 : PR마다 최소 1개 단위 테스트 + 1개 경계/에러 케이스 포함. 
3. 로그·검증 루틴 (동시 적용)
	- 행동 : 서비스 호출 전/후·변환 직후에 로그 남기기, null/빈 값 케이스 자동 검증 추가.
	- 목표 : 로컬에서 이상값 투입 테스트 자동화.
4. 입출력 명세·타임존·포맷 검증 (반드시)
	- 행동 : API 계약서(샘플 페이로드 포함) 작성, 날짜/타임존 표준 문서화.
	- 목표 : 배포 전 계약 불일치 0건 목표.
